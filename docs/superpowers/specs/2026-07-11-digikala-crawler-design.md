# Digikala Crawler — Design Spec

**Date:** 2026-07-11
**Status:** Approved, ready for implementation plan

## Context

China Import Manager already tracks purchases from Alibaba and inventory in Iran. The next commercial question is the *sell* side on Digikala: how our own listings are doing, and what competitors charge for equivalent products. `CLAUDE.md` already earmarks "a Digikala seller-product crawler" as a separate later phase — this spec designs it.

Two independent concerns, deliberately kept as separate subsystems (no foreign keys between them, own tables each):

1. **Own products** — monitor price / stock (and sales count when the panel exposes it) for the listings we sell on Digikala.
2. **Competitors** — track the price of equivalent competitor listings so we can price against them.

## Key technical finding (verified)

Unlike Alibaba (whose pages return an empty JS shell to a plain fetch, forcing the bookmarklet approach), **Digikala exposes a clean public JSON API** that needs no auth:

```
GET https://api.digikala.com/v2/product/<dkp-id>/
```

Verified live (2026-07-11) against several products. Relevant fields:

| Field | JSON path | Example |
|-------|-----------|---------|
| Title | `data.product.title_fa` | "قاشق چای خوری…" |
| Status | `data.product.status` | `marketable` / `out_of_stock` |
| Price | `data.product.default_variant.price.selling_price` | `6400000` |
| Seller | `data.product.default_variant.seller.title` | "نلین گالری" |

**Prices are returned in Rial**; the rest of the app is in Toman, so divide by 10 on store. Out-of-stock products omit `default_variant` — treat missing price as `null`, do not error.

Consequence: **competitor price *updates* can run entirely server-side** via this API, no browser needed. The bookmarklet is only used to *start* tracking a competitor (register URL + initial price). Own-product listings still need the bookmarklet because the seller panel is authenticated (public API doesn't expose our panel-only fields like sales count).

## Data model

`db/schema.sql` gains four tables (all `CREATE TABLE IF NOT EXISTS`, matching the existing idempotent-on-startup pattern).

### Own products (seller panel)

```sql
CREATE TABLE IF NOT EXISTS digikala_own_items (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  digikala_id   TEXT NOT NULL UNIQUE,   -- dkp product code from seller panel
  title         TEXT NOT NULL,
  product_id    INTEGER REFERENCES products(id) ON DELETE SET NULL,  -- manual link to a system product
  price         INTEGER,                -- Toman (Rial/10)
  stock         INTEGER,
  sales_count   INTEGER,                -- from panel if shown, else NULL
  captured_at   TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS digikala_own_price_history (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id     INTEGER NOT NULL REFERENCES digikala_own_items(id) ON DELETE CASCADE,
  price       INTEGER,
  stock       INTEGER,
  recorded_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Competitors

```sql
CREATE TABLE IF NOT EXISTS digikala_competitor_items (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  url          TEXT NOT NULL UNIQUE,   -- normalized product url
  digikala_id  TEXT,                   -- dkp id extracted from url, used for API refresh
  title        TEXT NOT NULL,
  seller_name  TEXT,
  price        INTEGER,                -- Toman
  status       TEXT NOT NULL DEFAULT 'ACTIVE' CHECK (status IN ('ACTIVE','IGNORED')),
  captured_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS digikala_competitor_price_history (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id     INTEGER NOT NULL REFERENCES digikala_competitor_items(id) ON DELETE CASCADE,
  price       INTEGER,
  recorded_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Both histories are **append-only, change-only** — a new row is inserted only when the parsed price actually differs from the last stored value, exactly mirroring `crawled_prices` for Alibaba (`helpers/crawl.js`).

## Data flow

### Competitor: initial capture (bookmarklet)

1. A new bookmarklet (generated by `services/bookmarklet.js`, targeting `digikala.com`) runs on a competitor's product page and extracts RAW strings only (title, price text, seller, current URL).
2. Posts to `POST /api/digikala/competitor/capture` — cross-origin from `*.digikala.com`, authenticated by the existing capture token (`helpers/settings.js`'s `getOrCreateCrawlToken()`, reused — not a new token) with the same manual CORS handling as `routes/crawl.js` (not the `cors` package). Cookies aren't sent cross-site, hence token auth.
3. Server extracts the dkp id from the URL (`helpers/parse.js`'s new `extractDkpId`), normalizes the URL, parses the Toman price (Persian digits → ASCII), upserts by normalized URL, and writes the first history row.

### Competitor: price refresh (manual per-item button, server-side)

1. Each competitor item in the UI has a "🔄 بروزرسانی قیمت" button → `POST /api/digikala/competitor/:id/refresh`.
2. Server fetches `https://api.digikala.com/v2/product/<digikala_id>/` itself (no browser), reads `selling_price` (Rial → ÷10), `status`, `seller.title`.
3. If the price changed, appends a history row; always refreshes `updated_at`. Out-of-stock → price kept `null`, item flagged, never deleted.

### Own products: capture (bookmarklet, bulk)

1. A second bookmarklet runs on the seller panel product-list page and reads ALL listing cards on the current page in one click (price / stock / sales count / dkp id / title), same "one click = whole page" model as the Alibaba crawler.
2. Posts to `POST /api/digikala/own/capture` (token auth, CORS). Server upserts each by `digikala_id`, appends change-only history.
3. `PATCH /api/digikala/own/:id/link` manually attaches an own-item to an existing `products.id`.

## Routes — `routes/digikala.js` (new)

```
POST  /api/digikala/competitor/capture       token+CORS, no session
POST  /api/digikala/competitor/:id/refresh    requireAuth, server-side API fetch
PATCH /api/digikala/competitor/:id/status     requireAuth, ACTIVE/IGNORED
GET   /api/digikala/competitor                requireAuth, list + latest price
POST  /api/digikala/own/capture               token+CORS, bulk
PATCH /api/digikala/own/:id/link              requireAuth, link to products.id
GET   /api/digikala/own                        requireAuth, list
```

Response convention follows the app: success is a bare `res.json(obj)`; failure is `res.status(4xx|5xx).json({ error: 'پیام فارسی' })`, checks inline in handlers. Mounted in `server.js` alongside the other route modules.

## Helpers & services

- `helpers/digikala.js` (new) — `captureCompetitor`, `refreshCompetitor`, `setCompetitorStatus`, `captureOwnItems`, `linkOwnItem`, plus the shared change-only history insert. Mirrors `helpers/crawl.js` structure; synchronous `better-sqlite3`, `db.transaction()` where a row + history must be atomic.
- `helpers/parse.js` (extend) — `extractDkpId(url)` (returns the dkp id, rejects non-digikala hosts like `normalizeAlibabaUrl` rejects non-alibaba), and a Toman price parser for Persian-digit strings (reuse/extend existing numeric parsing).
- `services/digikala.js` (new) — thin wrapper around the public product API: `fetchProduct(dkpId)` returning `{ title, status, priceToman, sellerName }` or throwing a typed error for a clean 502. Keeps network I/O out of the helpers so tests can stub it.
- `services/bookmarklet.js` (extend) — two new self-contained `javascript:` builders (competitor page, seller panel list), same inline-only constraint (page CSP can block external `<script src>` but not an inline bookmarklet).

## Frontend

New `public/digikala.html` + `public/js/digikala.js`, following the existing one-page-one-JS-file convention and reusing `public/js/api.js` (`api.get/post/patch` + `showToast`) and `public/js/header.js` (`initHeader`). Two independent tabs:

- **محصولات من** — own items, price/stock/sales, "link to product" action, price history toggle.
- **رقبا** — competitor items, per-item "🔄 بروزرسانی قیمت" button, ACTIVE/IGNORED toggle, price history toggle.

No automatic comparison table — the two tabs are independent lists (user compares mentally), matching the current `crawled.html` shape. A new nav item is added in `header.js`. Both bookmarklet install snippets are surfaced on this page (like the Alibaba "ذخیره در بار ابزار" affordance).

## Error handling

- Digikala API unreachable / timeout → `502 { error: 'خطا در دریافت از دیجی‌کالا' }` (distinct, not a stack trace), mirroring the navasan `MISSING_TOKEN`/`PARSE_ERROR` clean-error pattern.
- Product not marketable (`status != 'marketable'`) → price stored as `null`, item flagged "ناموجود", never deleted.
- Non-digikala URL in `extractDkpId` → rejected, capture entry skipped silently (batch survives), same as the malformed-entry handling in `captureItems`.

## Testing (node:test, in-memory DB via `CIM_DB_PATH=:memory:`)

- `extractDkpId`: valid dkp URLs, relative/absolute, rejects non-digikala and lookalike hosts.
- Toman price parser: Persian digits, thousands separators, garbage → null.
- Competitor upsert + change-only history: first capture creates item+price row; re-capture same price → no new history row; changed price → appended row.
- `refreshCompetitor` with the `services/digikala.js` fetch **stubbed** (no live network in tests): price change appends history; out-of-stock keeps null; API error surfaces as a typed error.
- Own capture bulk: multiple cards in one payload upserted by `digikala_id`; malformed entries skipped, batch survives.
- Status toggle rejects unknown id.

Live end-to-end (real bookmarklet on digikala.com in the user's browser, real seller panel) is handed off to the user, same as the Alibaba Phase 2 live step.

## Explicitly out of scope (YAGNI)

- Automatic own-vs-competitor comparison table or undercut alerts (user compares mentally for now).
- Scheduled/cron auto-refresh (manual per-item button only).
- Bulk "refresh all" button.
- Any write-back to Digikala (read-only throughout).
